### 1.3 设计原则 / Design Principles

- **单一职责**：每个文件/模块只做一件事（20 个源文件，最大不超过 400 行）
- **TDD 测试优先**：先写失败测试，再写实现代码。190 个单元测试，测试代码量 > 源码量（3575 行 vs 2569 行）
- **零依赖优先**：CJK 分词用 `Intl.Segmenter`（运行时内置），不引入第三方分词库
- **YAGNI**：只实现需求明确要求的功能，spec 是真理之源

## 2. 子代理规划实现 / Sub-Agent Planning

### 2.1 Agent Loop 核心流程

Agent Loop 采用 **AsyncGenerator** 模式，每轮 LLM 调用产生的中间事件（text、tool_call、status）通过 `yield` 实时推送给前端：

```
User Message
     │
     ▼
┌─────────────────────────────────────┐
│          Agent Loop (max 30 rounds) │
│                                     │
│  ┌───────────┐    has tool_calls?   │
│  │ LLM Call  │───────┬───────────▶  │
│  │ (SSE)     │       │ No: yield    │
│  └─────┬─────┘       │ text + break │
│        │             │              │
│   yield events       │ Yes:         │
│   (text/status/      │ dispatch all │
│    tool_call)        │ tools        │
│        │             │              │
│        ▼             ▼              │
│   ┌─────────────────────────┐       │
│   │ Tool Dispatch (batch)   │       │
│   │ • task_create/update    │       │
│   │ • web_search/fetch      │       │
│   │ • spawn_agent           │       │
│   │ • file_search           │       │
│   └───────────┬─────────────┘       │
│               │                     │
│         inject results              │
│         back into messages          │
│               │                     │
│               ▼                     │
│          next LLM round ────────────┘
└─────────────────────────────────────┘
     │
     ▼
  SSE Stream → Frontend
```

**关键设计**：一次 LLM 响应可能产生多个 tool_calls（如 5-8 个），全部并行 dispatch 后才进入下一轮 LLM 调用。

### 2.2 spawn_agent 子代理机制

当用户提出复杂研究主题时，主 Agent 通过 `spawn_agent` 工具将任务拆解为 1-3 个子任务，每个子任务启动独立的 Agent Loop 实例并行执行：

```typescript
// tools.ts — spawn_agent 定义
{
  name: 'spawn_agent',
  parameters: {
    tasks: { type: 'array', items: { type: 'string' }, minItems: 1, maxItems: 3 },
    context: { type: 'string', description: 'Shared context for sub-agents' }
  }
}

// loop.ts — 并行执行
const settled = await Promise.allSettled(
  tasks.map((task, i) => runSub(deps, `sub-${i}`, userId, task, ctx))
);
```

**子代理隔离**：
- 每个子代理有独立的 `agentId`（如 `sub-0`、`sub-1`）
- 使用精简版的工具集（排除 `spawn_agent`，防止无限递归）
- 心跳超时机制：10 秒内无 LLM 事件自动终止，防止子代理挂起

### 2.3 心跳超时机制

子代理没有固定超时，而是使用**心跳检测**：只要 LLM 持续产生事件（text、tool_call 等），就认为子代理仍在正常工作。超过 10 秒无事件才触发超时。

```typescript
// sub-agent.ts
const heartbeat = new Heartbeat(10_000);
for await (const event of agent.execute(...)) {
  heartbeat.ping();  // 收到任何事件就重置计时器
  yield event;
}
```

## 3. 记忆召回（RAG）设计 / RAG Pipeline

### 3.1 Hybrid Search 混合检索

系统采用 **FTS5 关键词搜索 + Qdrant 向量搜索 → RRF 融合 → MMR 去重** 的三阶段混合检索：

```
Query: "用户的问题"
     │
     ├──▶ FTS5 Keyword Search ──▶ [chunk1, chunk3, chunk5]
     │         │                            │
     │    CJK 分词              BM25 排名    │
     │                                  RRF 融合
     │                                    │
     └──▶ Qdrant Vector Search ──▶ [chunk2, chunk3, chunk7]
              │                            │
         bge-m3 向量化    余弦相似度        │
                                         RRF 融合
                                           │
                                     ┌─────▼─────┐
                                     │ RRF Fusion │
                                     │ k=60       │
                                     └─────┬─────┘
                                           │
                                     ┌─────▼─────┐
                                     │ MMR Rerank│
                                     │ λ=0.7     │
                                     │ topK=5    │
                                     └─────┬─────┘
                                           │
                                     Top 5 diverse results
                                     → 注入 LLM system prompt
```

**RRF (Reciprocal Rank Fusion)**：将两个排序列表融合为统一排序，公式：`score = Σ weight / (k + rank + 1)`，k=60。对排名靠前的结果给予额外奖励（rank 0: +0.05, rank 1-2: +0.02）。

**MMR (Maximal Marginal Relevance)**：在保持相关性的同时最大化多样性，避免返回内容高度重复的 chunk。公式：`MMR = λ × relevance - (1-λ) × max_similarity_to_selected`。

### 3.2 中文处理 / Chinese Text Processing

中文搜索的核心挑战是**分词**。FTS5 默认的 `unicode61` tokenizer 不理解中文词语边界，会将"人工智能"拆成单个字符。

**解决方案**：在 FTS5 查询前，用 `Intl.Segmenter`（浏览器/Worker 运行时内置，零依赖）进行 CJK 分词：

```typescript
// cjk.ts — 利用运行时内置 API
export function tokenizeCJK(text: string): string {
  const segmenter = new Intl.Segmenter('zh', { granularity: 'word' });
  const tokens: string[] = [];
  for (const { segment, isWordLike } of segmenter.segment(text)) {
    if (isWordLike || /[\u4e00-\u9fff]/.test(segment)) {
      tokens.push(segment);
    }
  }
  return tokens.join(' ');
}
```

- 输入："明天七点起床" → 输出："明天 七点 起床"
- FTS5 查询使用前缀匹配（`明天* AND 七点* AND 起床*`）
- 向量搜索天然支持中文（bge-m3 多语言模型）

### 3.3 Embedding 优化

同一 query 在 vector search 和 MMR reranking 中只调用一次 embedding API，避免重复调用导致成本翻倍：

```typescript
// search.ts — vectorSearchWithVectors 返回 queryVector 复用
const { results, queryVector } = await vectorSearchWithVectors(embedding, qdrant, query, userId);
// ... RRF fusion ...
const reranked = mmrRerank(candidates, queryVector, 0.7, 5);  // 复用 queryVector
```

## 4. 文件处理与向量化 / File Processing Pipeline

### 4.1 上传流程

```
用户上传文件 (PDF/DOCX/TXT/MD)
     │
     ▼
┌─────────┐   ┌─────────┐   ┌──────────┐   ┌──────────┐
│ R2 存储  │──▶│ Parser  │──▶│ Cleaner  │──▶│ Chunker  │
│ (原始文件)│   │ (解析文本)│   │ (清洗文本)│   │ (智能分块)│
└─────────┘   └─────────┘   └──────────┘   └──────┬───┘
                                                   │
                                              ┌────▼────┐
                                              │ D1 写入  │ ← chunk 元数据
                                              │ chunks   │   (doc_id, seq, content, token_count)
                                              │ chunks_fts│  (FTS5 同步写入)
                                              └────┬────┘
                                                   │
                                              ┌────▼────┐
                                              │ Outbox  │ ← 写入 pending 事件
                                              │ Event   │
                                              └────┬────┘
                                                   │
                                              ┌────▼────┐
                                              │ Embed   │ ← bge-m3 向量化
                                              │ + Upsert│ ← Qdrant 写入
                                              └─────────┘
```

### 4.2 智能 Chunker 分块策略

采用 **heading-aware** 分块，尊重文档结构：

- 目标大小：~500 tokens
- 重叠率：15%（75 tokens overlap），确保跨块语义连续
- 标题边界：遇到 `#` 标题时强制切分，保持块内主题一致
- Token 估算：基于字符数的快速估算（中文 1 字 ≈ 1 token，英文 4 字符 ≈ 1 token）

### 4.3 文本清洗

`cleaner.ts` 处理：
- Unicode NFC 标准化（统一全角/半角字符）
- 移除控制字符
- 清理 HTML 标签
- 合并多余空白

### 4.4 Outbox 双写一致性

**问题**：文件处理需要同时写入 D1（chunk 元数据）和 Qdrant（向量索引），但这两个系统没有分布式事务。

**解决方案**：Outbox 模式——先写 D1（同步），再通过 Outbox 表异步写入 Qdrant：

```
1. 写入 D1 chunks + chunks_fts（事务 batch）
2. 写入 outbox_events（status=pending, chunk_id=N）
3. 定时任务（cron */5 * * * *）调用 /api/admin/process-outbox
4. 处理器：claimEvent（乐观锁）→ embed → Qdrant upsert → markCompleted
```

**乐观锁**：`claimEvent` 用 `UPDATE ... SET status='processing' WHERE status='pending' AND id=?` 抢占事件，`changes > 0` 才处理，防止并发重复处理。

**失败重试**：处理失败时 `attempts + 1`，达到 3 次后标记 `failed`。`pending` 事件有 30 秒冷却期，避免立即重试。

## 5. 挑战与解决方案 / Challenges & Solutions

### 5.1 多数据库一致性 / Multi-Database Consistency

**挑战**：D1（SQLite）和 Qdrant（向量数据库）是两个独立的存储系统，无法使用传统事务保证一致性。如果 Qdrant 写入失败，D1 中已有 chunk 数据但无法被搜索到。

**解决方案**：Outbox 模式 + 定时重试 + 乐观锁：

| 组件 | 职责 |
|------|------|
| `outbox_events` 表 | 记录所有待处理的向量化事件 |
| `claimEvent()` | 乐观锁防止并发重复处理 |
| `markFailed()` | 失败重试（最多 3 次），退避 30 秒 |
| Cron (*/5 min) | 定时触发处理，最终一致性 |

这种模式参考了分布式系统中的 [Transactional Outbox](https://microservices.io/patterns/data/transactional-outbox.html) 模式，在不引入消息队列的情况下实现了最终一致性。

### 5.2 中文搜索质量 / Chinese Search Quality

**挑战**：FTS5 默认 tokenizer（`unicode61`）不理解中文词语边界。例如"人工智能"被拆成"人"、"工"、"智"、"能"四个单字，导致搜索"智能"时也能匹配到"人工智能"——看起来是对的，但搜索"工人"时也会匹配到，这就错了。

**解决方案**：两层结合：

1. **FTS5 关键词层**：用 `Intl.Segmenter` 预分词，将中文文本转为空格分隔的词语再存入 FTS5。搜索时同样先分词再查。
2. **向量语义层**：bge-m3 模型天然支持多语言语义理解，弥补关键词搜索的不足。
3. **RRF 融合**：两层结果融合，互相补充。

### 5.3 TDD 测试驱动开发 / Test-Driven Development

**挑战**：Cloudflare Worker 的测试环境与 Node.js 不同，D1 是 SQLite 绑定而非独立服务，需要在测试中正确模拟。

**解决方案**：

- 使用 `cloudflare:test` 和 `@cloudflare/vitest-pool-workers` 进行测试
- D1 测试使用真实的 D1 绑定（`env.DB`），在内存 SQLite 中执行真实 SQL
- Mock 仅用于外部服务（LLM API、Qdrant HTTP、Serper API）
- 严格遵守 TDD 流程：先写失败测试 → 实现 → 测试通过

**测试规模**：17 个测试文件，190 个测试用例，测试代码 3575 行（源码 2569 行，测试/源码比 = 1.39:1）

### 5.4 Agent Loop 可靠性 / Agent Loop Reliability

**挑战**：LLM 返回的 tool_call 参数可能是非法 JSON，静默吞错会导致后续逻辑使用空参数（如 `task_delete` 无 `id`）。

**解决方案**：
- LLM Client 层：`JSON.parse` 失败时，args 设为 `{ _parseError: "LLM returned invalid JSON..." }`，而非空对象
- Agent Loop 层：通过 `toolArgs` Map 直接传递解析后的 args 对象，消除不必要的 `JSON.stringify → JSON.parse` 往返
- 子代理：心跳超时（10 秒无事件自动终止），防止无限挂起
